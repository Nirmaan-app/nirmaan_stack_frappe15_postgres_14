01Where this lives — the data flow

After upload, each sheet gets a spoke screen at /upload-boq/hub/:boqId/sheet/:sheetName. The user configures the sheet, marks it "Config Done", then the hub runs a parse. Header & skip settings are set in the spoke but only take effect later, in the parser.

# Config is authored here, consumed much later. Three hops:

  SheetSpokePage.tsx  ──fetch preview──►  get_sheet_preview (sheet_preview.py)
        │                          raw cells, 40-row window, NO skip annotation
        ▼
  SheetConfigPanel.tsx  Section 1 (Rows) · 2 (Areas) · 3 (Column roles) · 4 (Work pkgs)
        │   header_row, header_row_count, top_header_rows_override,
        │   skip_top_rows_after_header, area_dimensions, column_role_map
        ▼  (Save / Mark Config Done)
  set_sheet_config (update_sheet_draft.py)  ─►  BoQ Sheet Draft.sheet_config  # JSON blob
        ▼  (later — hub "Parse")
  parse_run.py  ─►  orchestrator.parse_boq()  ─►  reader · classifier · hierarchy
        ▼
  BoQ Review Row[]   # what the user finally sees on the review screen
The single most important UX fact The preview grid the user stares at while configuring (SheetDataGrid.tsx) renders raw cells only. It highlights the header band (frozen rows) but shows no indication of: which rows will be auto-dropped, which rows the Skip field targets, or even a "data starts here" line in the grid itself. Every skip decision is authored blind and only becomes visible after a full parse.

The header pre-fill is seeded once, at upload, by detect_header_row() + auto_guess_sheet_config() (reader.py:356, _auto_guess.py:133). Critically, auto-guess sets header_row, header_row_count, column_role_map and area_dimensions — but never skip_top_rows_after_header. So the Skip field is always blank by default, with no hint about when it should be used.

02The header model — the part that "works" (mostly)

Section 1 of the config panel exposes four header controls:

Top header row: Yes / No → specifytop_header_rows_override (list|null)where the other header row is, when not adjacent

Single-header (header_row_count = 1) — fully consistent ✅

Header at row N → parser skips {N}, keeps rows ≥ N that aren't skipped → data from N+1. The UI's "Data starts at row N+1", the frozen-row highlight on row N, and the backend all agree. This is the common case, and it is clean — which is almost certainly why you experience header selection as "working correctly".

Double-header (header_row_count = 2) — the model fractures ⚠️

Here the same number header_row is interpreted two different ways inside the backend:

Logictreats header_row as…reads / skipscite
Row skipping the top of a 2-row band skips header_row and header_row + 1 (row below) orchestrator.py:174–176
Multi-area detection the bottom of a 2-row band reads header_row - 1 (row above) for area names orchestrator.py:242–246
Frontend label the bottom ("bottom header row"; "Top header will be row header_row-1") — SheetConfigPanel.tsx:891,955–958

Two of the three say header_row is the bottom row (with the second header above it). But the skip logic says it's the top (and skips the row below). These cannot all be right. Worked example for the most common 2-row layout — area names stacked above column labels:

With the natural input header_row = 7 (the column-label row, exactly what the UI label "bottom header row" and detect_header_row both point at) and header_row_count = 2:

# orchestrator.py:172-193 (verbatim behaviour)
skip_rows = {7, 8}            # header_row=7  AND  header_row+1=8
keep rows where (row >= 7) and (row not in skip_rows)
   row 6  ✗ dropped by (row >= 7)        # area banner — correct to drop
   row 7  ✗ skipped (header)              # correct
   row 8  ✗ skipped (header_row+1)      # ← BUG: this is the first DATA row
   row 9+ ✓ kept
# Meanwhile area detection reads row 6 (header_row-1) correctly. So the two halves disagree.

The user is forced into two bad choices, neither matches the sheet
  • Set header_row = 2 (the real column-label row): UI says "data starts row 4", but rows 4–6 (floors, towers, blank) then leak past the header and are silently auto-classified as spacers/junk. "Data starts" is wrong by 3 rows, and area detection (which reads header_row−1 = row 1, the title) finds nothing — so areas must be hand-typed (and were typed wrong).
  • Set header_row = 5 (the towers row): now 5 + 2 = 7 and "data starts row 7" is right, skip {5,6} removes towers + blank, and area detection reads bottom=row 5 (towers) / top=row 4 (floors) — which actually works. But calling row 5 the "bottom header row" is semantically absurd to anyone looking at the sheet (the column headers are obviously on row 2), and the column-label row 2 is then silently dropped by the ≥ header_row guard. It "works" only by gaming the number.

Three root causes (all one knot)

  1. The header model is capped at 2 rows. There is no way to say "the header spans rows 2–5". Real BoQs routinely stack column-labels + a supply/install split + one or two area tiers.
  2. "Data starts at row" is a naïve formula (header_row + header_row_count, SheetConfigPanel.tsx:469–470). It assumes data begins immediately after exactly header_row_count contiguous header rows. It ignores 3-4 tier headers, blank spacer rows, and title rows — so it is wrong whenever the sheet isn't the simple case.
  3. header_row is overloaded with three jobs that conflict on multi-tier sheets: (a) mark where column labels are, (b) define the skip band {header_row … header_row+hrc-1}, (c) seed area detection at header_row−1. One number can't satisfy all three when the header has >2 tiers — which is exactly why "data starts" is wrong and the areas come out mis-mapped.
This is also the "auto-skip" you meant The rows between the nominal header and the real data (floors, towers, blank — rows 4–6 here) don't get dropped; they get silently re-labelled as spacers/junk by the classifier. That invisible re-labelling, happening in the gap the header model can't describe, is the "automatic skipping" that feels opaque. Fixing the header model (so the header band actually covers rows 2–5) removes the gap, and the confusion with it.

03There are two completely different kinds of "skip"

This is, I think, the root of the conceptual confusion. The word "skip" is used for two mechanisms that live in different layers and behave differently:

layer A  Hard row-filter (a true drop)

Rows are removed from the row list before classification. They never exist downstream — not as a line item, not as a spacer, nothing.

  • The header row(s)
  • The user's skip_top_rows_after_header
  • Everything above header_row (the ≥ header_row guard)

Lives in orchestrator.py:189–193.

layer B  Soft classification (kept, but not data)

Rows stay in the output but get tagged a non-line-item type. They do appear on the review screen (as spacers / notes / subtotals), they're just not priced.

  • Blank rows → SPACER
  • Repeated header rows → HEADER_REPEAT
  • Subtotal / "Grand Total" rows → SUBTOTAL_MARKER
  • Empty "ghost" notes / junk-trio → SPACER

Lives in classifier.py (see §05).

Plain-English takeaway The Skip field is the only "hard drop" the user controls. Everything the user probably thinks of as "rows being skipped automatically" (blanks, subtotals, repeated headers) is actually Layer B — those rows are not dropped, they're re-labelled and still shown. So the field and the auto-behaviour aren't even the same operation, which makes "do I need to list this row in Skip?" genuinely hard to reason about.

04The user "Skip rows after header" field

The field (SheetConfigPanel.tsx:998–1017) is a free-text box in Section 1:

Label: "Skip rows after header (situational)" Placeholder: "e.g. 8, 9 (usually blank)"

Help text: "Comma-separated absolute row numbers to exclude. Non-numeric entries are ignored."

What it actually does

Input is parsed by parseIntList (:203–208): split on commas, parseInt, keep only n > 0. The resulting list is stored as skip_top_rows_after_header and, at parse time, dumped straight into the skip set (orchestrator.py:177):

skip_rows.update(sheet_config.skip_top_rows_after_header)
# then, the filter:
keep row iff  (row not in skip_rows)  and  (row >= header_row)

So the values are absolute 1-based Excel row numbers (matching the help text). Confirmed against the backend. Good — but the surrounding design produces at least five distinct ways to confuse a user:

#ConfusionWhy
C1 Name says "after header", behaviour says "absolute" The field key (skip_top_rows_after_header) and label both imply offsets relative to the header ("skip the 1st & 2nd rows after the header"). The help text + backend say absolute Excel rows. A user who types 1, 2 meaning "first two rows after header" actually targets Excel rows 1–2.
C2 Anything below header_row silently does nothing The row >= header_row guard means a skip value less than the header row is a no-op. If header is row 7 and the user enters 3, nothing happens — no error, no hint. (And C1 makes entering a too-small number likely.)
C3 Zero preview feedback The grid never highlights the rows you typed. You enter numbers, save, run a full parse, then go to the review screen to discover whether you guessed right. There is no "these rows will be excluded" overlay.
C4 Mostly redundant with auto-skip Blank rows, repeated headers and subtotals are already handled (Layer B). The only case the field uniquely covers is a content-bearing, non-header-like junk row right after the header (e.g. a "Project: ABC" banner, a units legend). For a clean BoQ it should always be blank — yet it sits prominently in Section 1.
C5 It's the wrong tool for the row it targets Its one legitimate use (a stray junk row) is also fixable later, non-destructively, on the review screen (hide / reclassify a row) — which is visual and reversible. The blind up-front textbox forces the decision at the least-informed moment.

05Automatic skipping at parse time

Beyond the header rows, the parser drops or re-labels rows on its own. Full enumeration, with the drop-vs-relabel distinction made explicit (this is the table I'd want pinned on the wall):

There are also several reclassification post-passes (PREAMBLE↔LINE_ITEM demotion/promotion, section-header promotion) — they change a row's type but never drop it, so they're out of scope for "skipping" but worth knowing they exist (they explain why a row's classification on the review screen may not match naïve expectation).

How detect_header_row() seeds the pre-fill

(reader.py:356) Scans the first ~15 rows, scoring each by header keywords — strong (sl.no, description) = +2/cell, medium (item, materials) = +1/cell. Guards skip rows with <3 non-empty cells, rows with a long (>60-char) narrative cell, and the last content row. Highest score wins (earliest on a tie); needs score > 1 or returns None. Because it keys on column-label keywords, in a 2-row header it lands on the bottom (column-label) row — which is exactly what feeds the §02 inconsistency.

06How the two skips overlap (redundancy)

  • They can target the same row harmlessly. If a user lists row 8 in the Skip field and row 8 is also blank, it's dropped by Layer A; Layer B never sees it. No double-processing, no error — just redundant.
  • No ordering dependency. Layer A runs first (rows removed before classification), so Layer B only ever sees survivors. The user's field always "wins" by virtue of running first.
  • The field is strictly weaker than it looks. Its only non-redundant power is dropping a content-bearing row that the classifier would otherwise treat as a real line/preamble/note. Everything else it could target is already handled automatically.

07Findings — what looks wrong, unnecessary, or confusing

Ranked by my confidence. Tags: confirmed = read directly in code; verify = needs a real-sheet test; design = a UX/clarity judgement.

F0  confirmed · live  The header model can't describe real multi-tier headers → wrong "Data starts" + mis-mapped areas

The headline defect (see case study). The 1–2 row header cap, the naïve header_row + header_row_count "data starts" formula, and the triple-overloaded header_row together make 3-4 tier sheets (HVAC Low side, and likely most multi-area BoQs) impossible to configure correctly. Not a data-drop bug, but the source of the "data starts is wrong sometimes" + "auto-skip is confusing" experience. This is the fix target for this branch.

F1  not reproduced  2-row header "drops first data row" — verify tested, no data loss

The header_row + 1 skip is a real logical fragility (skip-math treats header_row as the band top; area-detection + UI labels treat it as the bottom). But the live-data sweep (§02) shows it does not cause data loss on any real sheet — users set header_row at the band top, so header_row+1 is a genuine second header. Downgraded from "highest-impact" to a documentation/robustness note. Worth a defensive guard + clearer 2-row labelling later, but it is not the row-dropping bug you were worried about.

F2  confirmed  The Skip field is authored blind

No preview feedback (C3), no validation that a value is within the sheet / above the header (C2), no echo of which rows it removed. Combined with the misleading "after header" naming (C1), it's a guess-and-check control in the least-informed spot in the flow.

F3  design  The Skip field is largely unnecessary

For clean BoQs it should always be blank; its one real use (a content-bearing junk row) overlaps with the review screen's row-hide. Candidate outcomes: (a) demote it (collapse under an "Advanced" disclosure, relabel to "Exclude specific Excel rows", validate & preview), or (b) remove it and lean on auto-skip + review-screen hide. I'd lean (a) until we confirm no real sheet depends on it.

F4  design  "Skip" is overloaded across two layers

The hard-drop field and the soft auto-classification share the word "skip" but are different operations (§03). Users can't tell whether a blank/subtotal row "needs" listing in the field (it doesn't). Clarifying the vocabulary — "Exclude" (drop) vs "auto-detected non-data" (relabel) — would remove a lot of the fog.

F5  design  top_header_rows_override only feeds area detection

The "Top header row: Yes / No → specify" control and its < header_row validation exist solely to point area-detection at the right row (orchestrator.py:242–246); it plays no part in skipping. On a single-area 2-row sheet it does nothing useful yet still adds a decision. Worth scoping down or hiding unless the sheet is multi-area.

F6  not a bug  Single-header path is consistent

Recording the good news: for header_row_count = 1, the UI hint, the frozen-row highlight, and the backend skip all agree. No change needed there; the problems are concentrated in the 2-row path and the Skip field.

08Open questions & what I'd verify next

Before proposing fixes, a few things I'd like your read on (and one thing I'd test):

Q1 — Intent of the Skip field. Do you have (or can you recall) a real sheet where a user genuinely needed to type into "Skip rows after header"? If it's never legitimately used in practice, F3-option-(b) "remove it" becomes viable.
